# 网页端模拟展示技术方案：智能骑行眼镜 + 后方雷达

## 一、方案目标

在当前没有实际硬件原型的前提下，通过网页端构建一个具备前后端完整架构的模拟系统，用于展示“智能骑行眼镜 + 后方雷达”的核心能力。

该方案的目标不是只做一个演示动画，而是做一个具备系统闭环的比赛级原型，能够体现：

- 后方车辆感知与风险预警能力
- 智能眼镜的人机交互能力
- 医疗场景下的用户分层与风险管理能力
- 前后端联动、数据存储、报告生成和医生端查看能力

---

## 二、为什么要做前后端完整架构

如果只有前端页面，评委容易认为这只是一个视觉 Demo，而不是一个可落地的产品系统。

采用前后端分层方案后，可以展示以下工程能力：

- 前端负责道路场景、眼镜提示、医生端面板等可视化
- 后端负责风险计算、事件流处理、数据存储和报告生成
- 数据层负责用户档案、设备状态、风险事件和骑行报告沉淀
- 后续可以从模拟数据平滑升级到真实硬件接入

因此，整个系统可以被定义为：

**一个面向康复骑行场景的数字孪生演示平台。**

---

## 三、系统总体架构

建议系统采用前后端分离架构，整体分为六层：

1. 设备模拟与接入层
2. 后端网关层
3. 实时风险计算层
4. 业务服务层
5. 数据存储层
6. 前端展示层

### 1. 设备模拟与接入层

负责产生和接收与设备相关的数据输入。

当前比赛阶段：

- 由网页模拟器生成后方车辆数据
- 由后端模拟器生成雷达、眼镜、手机状态数据
- 由用户手动触发危险事件

后续硬件阶段：

- 接入毫米波雷达数据
- 接入手机蓝牙/传感器数据
- 接入眼镜设备状态数据

### 2. 后端网关层

负责统一接收前端和模拟器数据，对外提供 API 和实时连接能力。

主要职责：

- 提供 REST API
- 提供 WebSocket 通道
- 接收模拟器上传的传感器数据
- 管理用户身份、设备状态和会话状态

### 3. 实时风险计算层

负责对接收到的数据进行实时分析与风险分级。

主要职责：

- 计算目标距离和相对速度
- 计算 Time-to-Collision
- 根据用户档案进行个体化阈值判断
- 输出低、中、高风险等级
- 生成眼镜端预警指令

### 4. 业务服务层

负责医疗和产品相关的业务逻辑。

主要职责：

- 用户档案管理
- 康复人群分类管理
- 骑行会话管理
- 风险事件管理
- 康复安全报告生成
- 医生端查看和调整建议

### 5. 数据存储层

负责保存用户、设备、会话、事件和报告数据。

### 6. 前端展示层

负责以图形化方式展示整套系统的运行过程。

包括：

- 路演演示端
- 患者骑行端
- 医生管理端

## 四、推荐技术栈

## 4.1 前端技术栈

建议采用：

- React
- Vite
- TypeScript
- Tailwind CSS 或 CSS Modules
- Canvas 或 SVG
- ECharts / Recharts
- WebSocket Client

前端分为三个主要页面：

### 1. 路演演示页

用于比赛现场展示设备联动逻辑，包括：

- 骑行道路动态
- 后方车辆接近动画
- 眼镜风险提示
- 实时数据流

### 2. 患者端页面

用于展示个人骑行数据，包括：

- 本次骑行状态
- 风险事件数量
- 安全评分
- 历史记录

### 3. 医生端页面

用于展示医疗场景价值，包括：

- 患者档案
- 骑行报告
- 风险事件时间线
- 阈值调整

## 4.2 后端技术栈

建议采用：

- Node.js
- NestJS
- TypeScript
- RESTful API
- WebSocket

选择 NestJS 的原因：

- 模块化结构适合比赛答辩时讲架构
- 与 TypeScript 和前端更容易统一
- 天然适合拆分控制器、服务、网关和任务模块

## 4.3 数据与中间件

建议采用：

- PostgreSQL：主数据库
- Redis：实时状态缓存
- BullMQ：异步任务队列

功能分工：

- PostgreSQL 存储用户档案、骑行会话、风险事件、报告
- Redis 存储当前设备状态、当前风险等级和实时会话数据
- BullMQ 处理报告生成、通知推送、事件归档

## 六、后端功能逻辑设计

后端是这套方案复杂度的核心。比赛中要突出的是：前端负责看得见的演示，后端负责真正的逻辑闭环。

## 6.1 用户档案与医疗标签管理

后端应维护用户档案，而不是只在前端做简单选项切换。

用户档案字段建议包括：

- userId
- name
- age
- gender
- conditionType
- rehabStage
- riskSensitivity
- recommendedDuration
- maxSafeHeartRate

用途：

- 定义不同康复人群的预警阈值
- 支撑个体化风险策略
- 为医生端和报告生成提供基础数据

## 6.2 骑行会话管理

每一次演示或真实骑行都可以抽象为一个 Ride Session。

后端应支持：

- 创建会话
- 开始会话
- 暂停会话
- 结束会话
- 查询会话详情

会话中记录：

- 用户信息
- 场景类型
- 开始时间
- 结束时间
- 累计风险事件数
- 平均风险值
- 最终安全评分

## 6.3 设备状态管理

后端维护统一的设备状态中心。

状态字段建议包括：

- radarOnline
- glassesOnline
- appOnline
- batteryLevel
- signalStrength
- lastHeartbeat

这样可以模拟“雷达在线、眼镜连接、手机同步正常”的设备联动感。

## 6.4 实时风险计算服务

后端需接收来自模拟器的目标数据，并实时完成以下计算：

- 距离判断
- 相对速度判断
- 逼近趋势判断
- TTC 计算
- 风险评分计算
- 个体化阈值映射

输出结果包括：

- 当前风险等级
- 风险解释原因
- 眼镜预警指令
- 是否需要记录风险事件

可采用简化规则：

风险评分 = 距离权重 + 相对速度权重 + 逼近趋势权重 + 用户脆弱性权重

风险等级映射为：

- 绿色：安全
- 黄色：警惕
- 红色：危险

## 6.5 事件流处理

建议后端通过 WebSocket 向前端推送实时事件，而不是让前端自己完全计算。

事件类型包括：

- `target_detected`
- `target_updated`
- `risk_level_changed`
- `alert_triggered`
- `device_status_changed`
- `session_finished`

意义：

- 前端能实时刷新 UI
- 医生端和患者端可以共享同一套事件流
- 架构更像真实在线系统

## 6.6 风险事件落库

当风险达到中高等级时，后端自动写入风险事件表。

建议记录字段：

- eventId
- sessionId
- eventType
- riskLevel
- distance
- relativeSpeed
- ttc
- direction
- reason
- createdAt

用途：

- 支撑报告生成
- 支撑医生端查看
- 支撑演示结束后的事件回放

## 6.7 报告生成服务

骑行结束后，后端自动生成一份“康复骑行安全报告”。

报告内容包括：

- 本次骑行总时长
- 风险事件数量
- 高风险事件次数
- 最危险时刻
- 风险原因总结
- 适合的路线建议
- 是否建议降低训练强度

报告生成建议走异步任务：

- 会话结束后投递任务
- 后台聚合事件和用户信息
- 生成结构化报告
- 前端轮询或订阅报告状态

## 6.8 医生端管理逻辑

医生端除了展示，还需要后端支撑以下逻辑：

- 查看患者最近骑行记录
- 查看某次高风险事件详情
- 调整患者风险阈值
- 标记患者为“户外骑行需谨慎”
- 导出康复安全报告

## 6.9 通知与告警逻辑

为了增强系统复杂度，可以加入通知模块。

触发条件例如：

- 连续 3 次高风险事件
- 单次极高风险事件
- 设备掉线
- 骑行异常中断

通知对象包括：

- 患者本人
- 家属端
- 医生端

比赛中即使不接真实短信，也可以在页面上模拟：

- 家属收到推送
- 医生后台收到告警
- 系统生成应急建议

## 七、后方车辆模拟逻辑设计

网页端和后端都可以参与模拟数据生成。

## 7.1 目标对象结构

每个后方车辆目标包含：

- id
- distance
- relativeSpeed
- direction
- type
- dangerousOvertake

例如：

```json
{
  "id": "car_01",
  "distance": 12.5,
  "relativeSpeed": 18,
  "direction": "rear_left",
  "type": "car",
  "dangerousOvertake": false
}
```

## 7.2 数据更新机制

系统可以每 200ms 到 500ms 更新一次目标数据，用于形成接近动画和实时风险变化。

数据生成方式可分两种：

### 方式 A：前端本地模拟

适合最简单原型。

### 方式 B：后端模拟器生成

由后端定时生成目标变化并推送给前端，更适合展示“服务端在处理实时数据”。

建议比赛中采用方式 B。

## 7.3 医疗用户差异化策略

支持不同用户档案：

- 普通用户
- 老年用户
- 心肺康复用户
- 骨科术后用户

差异化策略包括：

- 老年用户更早触发中高风险预警
- 心肺康复用户更强调高速度逼近风险
- 骨科术后用户更重视转头困难与偏航风险

## 八、网页端功能设计

## 8.1 演示控制功能

支持：

- 开始演示
- 暂停演示
- 重新开始
- 切换用户类型
- 切换场景
- 手动触发危险超车

## 8.2 场景切换

建议至少设计三类场景：

### 场景 1：城市慢行道路

- 车流较多
- 风险事件较频繁

### 场景 2：康复绿道

- 车流较少
- 风险较低

### 场景 3：混行道路

- 机动车与电动车混行
- 突发风险明显

## 8.3 眼镜提示模拟

由于网页无法真实震动和骨传导，建议使用等效表达：

- 震动提示：镜腿抖动动画
- 骨传导提示：语音文本气泡
- LED 光带：镜片边缘动态发光
- HUD 提示：箭头和风险等级图标

## 8.4 数据记录与回放

支持两种展示方式：

- 实时演示
- 演示结束后的事件时间线回放

回放内容包括：

- 来车时间点
- 风险上升过程
- 眼镜预警状态变化
- 最终事件记录

## 十一、系统时序逻辑

一次完整演示的时序建议为：

1. 前端选择用户类型和路况场景
2. 前端调用后端创建骑行会话
3. 后端返回 sessionId 和风险阈值配置
4. 模拟器持续向后端发送目标与设备数据
5. 后端实时计算风险并缓存当前状态
6. 后端通过 WebSocket 推送风险变化和预警事件
7. 前端更新道路视图、眼镜视图和设备状态
8. 中高风险事件自动写入数据库
9. 会话结束后，后端异步生成康复骑行安全报告
10. 医生端读取报告和事件时间线进行分析展示

这个时序能很好地体现系统复杂度和工程完整性。

## 十二、如何体现“硬件感”

网页演示最大的问题是容易被看成普通网页，因此要强化“设备联动”和“实时感知”。

建议：

### 1. 明确展示设备拓扑

页面中显示：

- 车尾雷达负责采集
- 眼镜负责提醒
- 手机负责记录
- 医生端负责分析

### 2. 加入设备状态标签

例如：

- 雷达在线
- 眼镜已连接
- 蓝牙正常
- 电量 82%

### 3. 增加实时数据流视觉语言

使用：

- 脉冲点
- 数据跳动数字
- 信号线动画
- 风险波形条

### 4. 加入多视角切换

支持切换：

- 道路视角
- 雷达视角
- 眼镜视角
- 医生视角

## 十三、与真实硬件对接的预留设计

虽然当前阶段使用网页模拟，但系统协议应按真实设备对接方式设计。

### 当前输入源

- 前端模拟器
- 后端模拟器
- 用户手动触发事件

### 后续可替换输入源

- 蓝牙串口数据
- Wi-Fi 设备数据
- 毫米波雷达串流数据
- 手机 IMU / GPS 数据

建议统一数据结构：

```json
{
  "timestamp": 1710000000,
  "sessionId": "session_001",
  "targets": [
    {
      "distance": 7.8,
      "relativeSpeed": 24,
      "direction": "rear_right"
    }
  ],
  "deviceState": {
    "radarOnline": true,
    "glassesOnline": true
  },
  "userProfile": "rehab_cardiac"
}
```

这样未来替换的只是输入源，不需要重写上层前后端逻辑。

## 十四、比赛级 MVP 建议

如果要兼顾复杂度和落地性，建议先做最小闭环版本：

- 前端 React 演示页面
- 后端 NestJS 服务
- WebSocket 实时事件推送
- PostgreSQL 数据库
- 一个风险计算模块
- 一个报告生成模块

这样已经足够支撑“完整前后端系统”的答辩叙事。

## 十五、方案优势

### 1. 不依赖真实硬件也能完整演示

可以在比赛阶段快速形成系统级原型。

### 2. 体现完整工程能力

不仅有界面展示，还有接口、服务、数据库和报告逻辑。

### 3. 更容易体现医疗属性

通过用户档案、医生端、风险事件和报告生成，项目更像医疗辅助系统而不是普通骑行设备。

### 4. 后续可平滑升级

当前使用模拟数据，未来可以直接接入真实雷达和眼镜设备。

## 十六、潜在限制与应对

### 限制 1：没有真实硬件

应对方式：
强调当前是“数字孪生演示系统”，用于验证产品逻辑和医疗场景价值。

### 限制 2：网页无法完全复现佩戴体验

应对方式：
通过第一视角 UI、实时事件流和眼镜预警动画尽量贴近真实场景。

### 限制 3：后端模块多，开发量较大

应对方式：
优先实现“模拟器 + 风险服务 + WebSocket + 报告生成”的最小闭环版本。

### 限制 4：算法目前偏规则驱动

应对方式：
说明比赛阶段先采用规则引擎，后续基于真实传感器数据训练模型。

---

## 十七、结论

在没有硬件原型的条件下，采用“前端可视化展示 + 后端风险服务 + 数据存储 + 医疗管理逻辑”的方式构建网页系统，是更适合比赛展示的方案。它不仅能展示“智能骑行眼镜 + 后方雷达”的感知与预警过程，还能体现完整的系统架构能力。

从技术路线看，建议先实现比赛级 MVP：由前端负责演示和交互，由后端负责会话管理、风险计算、事件落库和报告生成；后续再逐步接入真实雷达、眼镜和手机端数据，完成从概念验证到真实设备联动的升级。
